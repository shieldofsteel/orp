# Upgrading to ORP v0.3.0

v0.3.0 changes the on-disk wire format of two persistent stores. If you run
ORP with persistent state, follow the steps below before swapping the binary.

## What changed

- **Federation outbox** (`ORP_FED_OUTBOX_PATH`, RocksDB): values are now
  encoded with `bincode 2` (`config::standard()`, variable-int) instead of
  `bincode 1.3` (fixed-int). The wire format is **incompatible**.
- **Isolation Forest models** embedded via `include_bytes!` in `orp-ml`: same
  bincode 1 → 2 break. The model schema version was bumped from `1` to `2`
  so a stale embedded model surfaces as `MlError::ModelVersionMismatch`
  instead of silently mis-decoding. Anything you trained against v0.2.x
  must be retrained or re-serialised on v0.3.0.

The data event stream itself is unchanged — JSON over the wire, JSON in
DuckDB. Only the outbox-on-disk and embedded ML models are affected.

## Outbox migration

The v0.3.0 binary refuses to start against a v0.2.x outbox to prevent
mis-decoding bincode 1 bytes as bincode 2. You will see:

```
Refusing to start: FederationOutbox at /var/lib/orp/outbox is from an
incompatible (pre-v0.3.0) ORP release: wire-format marker is None,
expected b"v2". Drain the outbox with the v0.2.x binary before
upgrading. See docs/upgrades/v0.3.0.md.
```

### Recommended path: drain on v0.2.x first

1. **Stop the v0.2.x binary cleanly** so any in-flight work is flushed.
2. **Restart on v0.2.x** with peers reachable. The outbox flush loop
   delivers queued entries; let it run until pending counts reach zero
   (see `/healthz` or the federation metrics).
3. **Stop the v0.2.x binary**, then **delete the outbox directory**:
   ```
   rm -rf "$ORP_FED_OUTBOX_PATH"
   ```
4. **Start the v0.3.0 binary**. It will create a fresh outbox stamped with
   the new wire-format marker.

### Acceptable shortcut: discard the outbox

If you accept losing whatever is currently queued (e.g. dev environments,
or peers you are not coming back to), just delete the directory before
starting v0.3.0:

```
systemctl stop orp
rm -rf "$ORP_FED_OUTBOX_PATH"
systemctl start orp     # now running v0.3.0
```

The outbox is a buffer for delivery to *peers*; entries you have already
acknowledged into your local DuckDB are unaffected.

### Skip the gate (not recommended)

You can disable federation buffering for one session by unsetting
`ORP_FED_OUTBOX_PATH`. The startup gate fires only when the path is set.
This drops *new* federation work for the run; it does not migrate the
existing outbox.

## Isolation Forest models

If you `include_bytes!` a v0.2-era IF model:

1. Re-train with the v0.3.0 toolchain, or
2. Load it once with the v0.2.x binary and re-serialise:
   ```
   let model = IsolationForestModel::deserialize_v0_2(bytes)?;
   let new_bytes = model.to_bytes()?;          // v0.3.0 wire format
   std::fs::write("model.bin", new_bytes)?;
   ```

The embedded model bytes shipped in `orp-ml` itself are regenerated as
part of the release process; you only need to act if you ship your own.

## Why we did this

`bincode` is flagged unmaintained (RUSTSEC-2025-0141). We bumped to the
last released line (2.x) to get off the ancient 1.3 API surface and to
make a future migration to `postcard` a one-step rewrite rather than
two. The version-marker mechanic added here is reusable for the next hop.
